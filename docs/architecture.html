The Kernel — @torquedev/core

The kernel provides mechanisms, not policies. It resolves bundles, boots them, routes requests, enforces isolation, discovers Intents, and compiles them into dual-interface APIs and Agent Tool Schemas. It has zero business logic — no knowledge of domains or application-specific concepts. Business logic lives in bundles. Intent constraints live in bundles. The kernel enforces both.

Registry kernel

Bundle lifecycle manager — loads bundles in dependency order, wires interfaces, discovers Intents via instance.intents(), cross-references manifest declarations against implementations, and holds all booted bundle state including Intent runtime metadata.

package:@torquedev/core file:kernel/registry.js
Constructor:
new Registry({ dataLayer, eventBus, createScopedData, hookBus })
Key methods:
boot(mountPlanPath, { sorted, bundleDirs }) — load plan, boot in dep order
call(bundleName, interfaceName, args) — cross-bundle interface routing
activeBundles() — array of booted bundle names
bundleInstance(name) — the instantiated bundle class
bundleManifest(name) — the parsed manifest object
bundleDir(name) — filesystem path for a bundle
bundles[name].intents — registered Intent objects (runtime metadata)
Validation modes:
validation.contracts: warn (default) or strict
validation.events: warn (default) or strict
Strict mode throws ContractViolationError on mismatch
Interface call routing:
1. Look up handler in interface map
2. Run interface:gate hook (can abort by throwing)
3. Fire interface:before-call hook (observer)
4. Execute handler function
5. Validate return value vs. manifest output.shape
6. Fire interface:after-call hook
7. On error: fire interface:error, re-throw
Intent discovery:
1. Call instance.intents() on each booted bundle
2. Store Intent objects as runtime metadata in bundles[name].intents
3. Server auto-registers POST /api/intents/{bundle}/{intent}
4. HookBus emits idd:intent_received, idd:executing, idd:intent_invoked

Resolver kernel

Resolves bundle sources from git repos, local paths, or bare names. Handles ENV var interpolation, topological dependency sorting, circular dependency detection, and lock file management.

package:@torquedev/core file:kernel/resolver.js
Source types:
Bare name — checks bundles/<name>/, then .bundles/<name>/
Path — path:./relative/dir resolved from app root or plan dir
Git — git+https://... clones to .bundles/ cache; pinned via bundle.lock
ENV interpolation:
${VAR_NAME} — required, throws if unset
${?VAR_NAME} — optional, resolves to empty string
Static methods:
Resolver.interpolateEnv(yamlString)
Resolver.loadMountPlan(path) — loads + interpolates + parses
Circular dependency detection:
CircularDependencyError: Circular dependency detected: A → B → A
Boot aborts. All cycles must be resolved before the app can start.

HookBus kernel

Lifecycle hook system for audit logging, tracing, rate limiting, metrics, and custom middleware. Two categories: observers (fire-and-forget, errors swallowed) and gates (can abort execution by throwing).

package:@torquedev/core file:kernel/hooks.js
Bundle lifecycle:
bundle:before-boot / bundle:after-boot
Interface hooks:
interface:gate — throws to abort call
interface:before-call / interface:after-call / interface:error
Route hooks:
route:before / route:after / route:error
Event hooks:
event:before-publish / event:after-publish
Usage:
hookBus.register('interface:before-call', ctx => log(ctx))
hookBus.registerGate('interface:gate', ctx => checkAuth(ctx))

ScopedCoordinator kernel

Each bundle receives a ScopedCoordinator instead of direct registry access. Restricts cross-bundle calls to bundles declared in depends_on and optional_deps. Runtime enforcement — not a linter warning, a hard throw.

package:@torquedev/core file:kernel/coordinator.js
Constructor & API:
new ScopedCoordinator(registry, bundleName, allowedBundles)
async call(targetBundle, interfaceName, args = {})
On unauthorized call:
DependencyViolationError:
Bundle 'analytics' cannot call 'billing.getInvoice':
'billing' is not in depends_on or optional_deps.
Declared dependencies: [identity, pipeline]

Services — shared singletons

Three core services are provisioned at boot and injected into all bundles. Services are singletons — one instance per app. They coordinate bundle isolation, asynchronous messaging, and HTTP routing.

DataLayer service

SQLite via better-sqlite3. WAL mode for concurrent reads. Auto-provisions tables from manifest schemas. Per-bundle table isolation enforced at runtime — attempting cross-bundle table access throws BundleIsolationError.

package:@torquedev/datalayer
Isolation model:
Tables are prefixed {bundleName}_{tableName}. A bundle receives a BundleScopedData instance that filters all queries to its own tables. No direct SQL access.
BundleScopedData API:
insert(table, row) Insert one row, return last insert id
find(table, id) Find by primary key
query(table, where) Query with optional WHERE object
update(table, id, changes) Update by primary key
delete(table, id) Delete by primary key
count(table) Count rows in table

EventBus service

Pub/sub with durable persistence. Synchronous fan-out to all subscribers. In-memory ring buffer (200 events) + SQLite append log. Payload validation against manifest event schemas.

package:@torquedev/eventbus
Design principles:
Durable — persists to _kernel_events SQLite table
Ordered — events published in strict order
Synchronous — all subscribers execute before publish() returns
Validated — payload shape checked against manifest
API:
publish(eventName, payload) Publish and persist event
subscribe(eventName, handler) Subscribe to event, returns unsubscribe()
recentEvents(limit) Recent events from memory buffer
queryEvents(where) Query persisted events with WHERE clause
_kernel_events schema:
id, eventName, bundleName, payload (JSON), timestamp, sequence

Server service

Express HTTP server. Auto-registers all bundle routes from manifests at boot. Exposes system endpoints for health, introspection, and event history. Route handlers receive (req, res, { data, events, config, coordinator }).

package:@torquedev/server
Auto-wiring:
At boot, the Server reads routes[] from all active bundle manifests. Each route is registered as app[method](path, bundleHandler). Path and method come from manifest; handler is resolved from bundle logic.js.
System endpoints:
GET /health — server status
GET /api/manifest — all active bundle manifests
GET /api/events — recent event log
GET /api/introspect — bundle names, interfaces, routes
Route handler contract:
async handler(req, res, {
  data: BundleScopedData,
  events: EventBus,
  config: object,
  coordinator: ScopedCoordinator
})
Bundle Structure & Manifest Contract

Bundle File Tree

your-bundle/ ├── manifest.yml contract & schema ├── logic.js implementation + intents() ├── agent.md AI context (optional) ├── package.json npm metadata ├── test/ │ └── your-bundle.test.js ├── ui/ (optional) │ ├── index.js view routes │ └── ui-kit.js descriptor helpers └── intents/ (optional) ├── OrganizeWork/ │ ├── context.js data boundary + vectorize │ ├── behavior.js guardrails + allowed tools │ └── intent.js goal + success criteria └── SummarizeBoard/ ├── context.js ├── behavior.js └── intent.js

Manifest + Logic + Intents. The manifest declares tables, events, interfaces, and routes. The logic.js implements those contracts and exposes Intents via intents(). Each Intent is a triplet of Context (what data), Behavior (what guardrails), and Intent (what goal). The kernel discovers them at boot and auto-registers dual-interface APIs.

logic.js Skeleton

import { MyIntent } from './intents/MyIntent/intent.js';

export default class YourBundle {
  constructor({ data, events,
    config, coordinator }) {
    this.data = data;
    this.events = events;
    this.coordinator = coordinator;
  }

  // Intent-Driven — discovered at boot
  intents() {
    return { MyIntent };
  }

  // Cross-bundle interfaces
  interfaces() {
    return {
      greet: () => "hello",
    };
  }

  routes() { /* ... */ }
  setupSubscriptions(eb) { /* ... */ }
}

Three surfaces: intents() exposes Intent objects — compiled into dual-interface APIs automatically. interfaces() exposes cross-bundle calls via coordinator.call(). routes() exposes HTTP handlers. All three are discovered by the kernel at boot.

manifest.yml — Complete Example

name: your-bundle
depends_on:
  - auth-bundle
optional_deps:
  - analytics-bundle

config:
  schema:
    databaseUrl:
      type: string
      required: true
    apiKey:
      type: string
      required: false

tables:
  users:
    schema:
      id: INTEGER PRIMARY KEY
      name: TEXT NOT NULL
      email: TEXT UNIQUE

events:
  user:created:
    schema:
      userId: number
      email: string
  user:deleted:
    schema:
      userId: number

interfaces:
  getUser:
    input: { userId: number }
    output:
      shape: { id: number, name: string }
  createUser:
    input: { name: string, email: string }
    output:
      shape: { id: number }

routes:
  - method: GET
    path: /users/:id
    handler: getUser
  - method: POST
    path: /users
    handler: createUser

# Intents are declared in logic.js via intents(),
# not in the manifest. Each intent is a triplet:
#
# intents/ResolveIssue/
#   context.js    — Context({ schema, vectorize })
#   behavior.js   — Behavior({ persona, allowedTools,
#                     requireHumanConfirmation })
#   intent.js     — Intent({ name, trigger,
#                     successCriteria, behavior })
#
# The kernel auto-registers:
#   POST /api/intents/your-bundle/ResolveIssue
Manifest sections explained:
  • name — Bundle identifier (kebab-case)
  • depends_on — Required bundles; error if missing at boot
  • optional_deps — Optional bundles; coordinator.call() will error if not declared here
  • config.schema — Configuration fields and their types; kernel injects config satisfying this schema
  • tables — SQLite table definitions; prefixed with {bundleName}_{tableName} at boot
  • events — Event schemas this bundle publishes; validated by EventBus.publish()
  • interfaces — Functions other bundles can call via coordinator.call(); full input/output validation
  • routes — HTTP endpoints; auto-registered by Server at boot; method + path + handler name
  • intents() — (in logic.js, not manifest) Returns Intent objects; kernel auto-registers POST /api/intents/{bundle}/{intent} and compiles to Agent Tool Schemas
Boot Sequence — 11 Steps

The boot sequence is deterministic and linear. Each step must complete successfully before the next begins. Any error halts the boot and reports the failure.

1

CLI parses command

User runs torque start or node app.js. CLI resolves --mount-plan path and any override arguments (--port, --debug, etc.).

2

Load mount plan YAML

Resolver reads the mount plan file and interpolates ENV variables. Validates top-level YAML structure and bundles array.

3

Resolve bundle sources

For each bundle in the plan, Resolver determines its filesystem path: bare name (from bundles/ or .bundles/), local path (via path:../), or git URL (cloned to .bundles/ and locked in bundle.lock).

4

Topological sort by depends_on

Resolver builds the depends_on graph from all manifest files and performs a topological sort. If a cycle is detected, throws CircularDependencyError and halts.

5

Initialize shared services

Boot up DataLayer (SQLite), EventBus (in-memory ring buffer), and Server (Express). These are singletons — one per app.

6

Validate all manifests

Registry reads every manifest.yml and validates structure: required keys, schema types, no duplicate event names, no missing interface handlers. Validates depends_on references against active bundles. If validation.contracts: strict, enforces manifest↔implementation match.

7

Provision schemas (DataLayer)

For each bundle, DataLayer creates all SQLite tables declared in the manifest. Table names are prefixed {bundleName}_{tableName}. If WAL mode is not enabled, enable it now. Create _kernel_events table for event log.

8

Instantiate bundle classes with DI

For each bundle (in topological order), import its logic.js file, call the default export with { data, events, config, coordinator } (scoped to that bundle), and instantiate the bundle class.

9

Wire interfaces & subscriptions

Registry builds the interface map from all bundles. For each declared event subscription in logic.js, wire the handler to EventBus. Fire bundle:after-boot hook.

10

Discover and register intents

Registry calls instance.intents() on each booted bundle and stores the returned Intent objects as runtime metadata. The AgentRouter auto-registers POST /api/intents/{bundle}/{intent} for each discovered intent. HookBus emits intent_received, executing, and intent_invoked lifecycle events on every invocation.

11

Register HTTP routes & start server

Server registers all routes from all bundle manifests to the Express app. Start listening on --port (default 3000). Log "Server listening on port 3000".

Intent-Driven Architecture — the semantic action layer

An Intent is the atomic unit of meaning in Torque. Where interfaces are mechanical contracts between bundles, intents are semantic actions that carry purpose. Every intent is discoverable at boot, auto-routed by the kernel, and observable through lifecycle hooks. Intents manifest in three ways: as REST endpoints, as Agent Tool Schemas, and as Reactive Stream triggers.

Discovery at Boot intent

During boot (step 10), the Registry calls instance.intents() on every booted bundle. Each bundle returns an array of Intent objects declaring name, input schema, handler function, and human-readable description. The Registry stores these as runtime metadata alongside the interface map.

Intent declaration in logic.js:
export default async function TasksBundle({ data, events }) {
  return {
    // ... interfaces ...
    intents: () => [
      {
        name: 'createTask',
        description: 'Create a new task in a project',
        input: { title: 'string', projectId: 'number' },
        handler: async (args) => { /* ... */ }
      }
    ]
  };
}

AgentRouter intent

The AgentRouter is the kernel component that dispatches intent invocations. At boot, it reads all registered intents and auto-registers a POST /api/intents/{bundle}/{intent} route for each one. No manifest route declarations needed — intent routes are derived automatically.

Auto-registered route:
POST /api/intents/tasks/createTask
Request body is validated against the intent's input schema.
Dispatch flow:
1. Receive POST request at /api/intents/{bundle}/{intent}
2. Validate input against declared schema
3. HookBus emits intent_received
4. HookBus emits executing
5. Call intent handler with validated args
6. HookBus emits intent_invoked with result
7. Return result as JSON response

HookBus Lifecycle Events intent

Every intent invocation emits three lifecycle events through the HookBus. Gates can intercept intent_received to abort execution (e.g., authorization checks). Observers on intent_invoked enable audit logging, metrics, and reactive side effects.

Lifecycle events:
intent_received Fired before dispatch — gates can abort by throwing
executing Fired when the handler begins execution
intent_invoked Fired after handler completes — carries result payload
Hook context payload:
{ bundle, intent, args, result?, error?, timestamp }

Three Manifestations intent

Every declared intent automatically surfaces through three channels. No additional configuration or wiring is required — the kernel derives all three from the single intent declaration in intents().

1. REST Endpoint
POST /api/intents/{bundle}/{intent} — standard HTTP, JSON in/out, schema-validated. Human clients and external services consume intents here.
2. Agent Tool Schema
Each intent is exported as a tool schema at GET /api/intents/_tools. AI agents (Claude, GPT, local LLMs) consume this schema to discover and invoke intents programmatically. The schema includes name, description, and typed input parameters.
3. Reactive Streams
The HookBus lifecycle events (intent_received, executing, intent_invoked) allow any bundle to reactively subscribe to intent activity. This enables event-driven workflows, audit trails, and cross-bundle orchestration without direct coupling.

Key Insight: The Architectural Pattern Progression

Convention (Rails) made web apps fast to build, but relies on developer discipline. Compile-time (ASP.NET) enforced rules at build time, but creates friction. Runtime (Torque) validates composition, dependencies, and isolation at boot and at every call — catching errors early, not after deployment.

Torque is neither a convention-based framework nor a compile-time system. It's an intent-driven, declarative, runtime-validated composition engine that treats the application as a wired graph of isolated modules. The YAML file IS the app. Bundles are the units. Intents are the semantic actions. The kernel mediates all communication. This is the language-agnostic, agentic-first composable monolith framework.
