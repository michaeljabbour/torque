    Build a Task Manager from scratch

    Learn Torque by building a complete, beautiful task manager with React frontend, SQLite backend, cross-bundle events, middleware hooks, and full validation. In 30 minutes, you'll understand the entire framework.

    ⏱ 30 min read
    💾 Language-Agnostic, Agentic-First
    🎨 React Frontend
    ✦

    Torque Philosophy

    The ideas behind the framework

    Before writing code, it helps to understand why Torque works the way it does. Four principles drive every design decision.

    The LEGO Principle

    Small, self-contained bundles with a shared contract snap together. A LEGO brick doesn't care if it becomes a spaceship or a castle — the connector is the same on every piece. In Torque, the connector is manifest.yml. Every bundle declares what it provides and what it needs. The kernel reads these contracts and wires everything together at boot.

    Mechanism, Not Policy

    The kernel provides capability — bundles make decisions. Like the Linux kernel, Torque's core handles registration, isolation, and routing. It never contains business logic. The kernel reads your manifest, provisions your database, wires events, and enforces boundaries. Then it gets out of your way. This is the Unix philosophy applied to full-stack apps.

    Composition Scales Linearly

    Feature flags grow combinatorially — 10 flags with 2 states means 1,024 possible combinations to test. Bundles grow linearly — 10 bundles means 10 independent decisions. Each bundle is in or out, declared in a mount plan. No tangled conditionals, no if (featureX && !featureY) spaghetti. This is why composability wins at scale.

    Intent as First-Class Object

    Controllers tell a machine how to do something. Intents tell it what done looks like. You declare Context (data shape), Behavior (guardrails), and Intent (success criteria). The kernel compiles that into a REST endpoint, an Agent Tool Schema, and real-time WebSocket streams — simultaneously. Safety is mathematical: requireHumanConfirmation physically gates dangerous actions at runtime. You write constraints for an intelligence, not instructions for a machine.

    How bundles communicate

    Tasks
    publishes
    EventBus
    delivers
    Categories
    subscribes

    Bundles never import each other. They communicate through events and the coordinator.

    With these ideas in mind, let's build something real. By the end, you'll have a working app that demonstrates every concept above.

    Setup

    Create your Torque project

    Let's get started by installing Torque and creating your first project.

    Install Torque CLI

    Install the Torque CLI globally:

    bash 
    $ npm install -g @torquedev/cli

    Create your app

    Generate a new Torque project:

    bash 
    $ torque new task-manager
$ cd task-manager
$ npm install

    Explore the structure

    Your new project includes:

    task-manager/ ├── bundles/ # Your bundles go here ├── mount_plans/ # Environment configs ├── torque.config.js # Root configuration ├── package.json └── README.md
    📌
    What is a Bundle?
    A bundle is a self-contained module with its own database schema, event declarations, interfaces (HTTP routes), and business logic. Bundles communicate through events and a coordinator interface.
    2

    Your First Bundle

    Generate the tasks bundle scaffold

    Let's generate a bundle for managing tasks with a complete schema, events, and routes:

    bash 
    $ torque generate scaffold tasks title:string completed:boolean \\
  priority:string due_date:datetime

    This creates bundles/tasks/ with manifest and logic files. The manifest declares your schema, events, and interfaces.

    The Generated Manifest

    Your manifest.yml defines the contract:

    yaml bundles/tasks/manifest.yml 
    schema:
  task:
    id: uuid
    title: string
    completed: boolean
    priority: string
    due_date: datetime
    category_id: uuid

events:
  publishes:
    - task.created
    - task.updated
    - task.completed
  subscribes:
    - coordinator.boot

interfaces:
  - path: POST /tasks
    name: createTask
  - path: GET /tasks
    name: listTasks

    Boot and verify

    bash 
    $ torque dev
✓ tasks bundle loaded
✓ Coordinator ready
✓ Server listening on http://localhost:3000
    💡
    Database Included
    Torque bundles automatically use SQLite. Your schema is auto-migrated on startup. No separate database setup needed.

    Add a Frontend

    Build a React task manager UI

    Torque bundles can serve React frontends. Let's scaffold the views:

    bash 
    $ torque generate views tasks
✓ Created bundles/tasks/views/
✓ Created bundles/tasks/views/TaskManager.jsx

    React Task Manager Component

    Torque automatically compiles and serves React components. Build a beautiful task manager with filtering, priority badges, and a dark theme:

    javascript bundles/tasks/views/TaskManager.jsx 
    import React, { useState, useEffect } from "react";

export default function TaskManager() {
  const [tasks, setTasks] = useState([]);
  const [filter, setFilter] = useState("all");
  const [title, setTitle] = useState("");
  const [priority, setPriority] = useState("medium");

  useEffect(() => { fetchTasks(); }, []);

  const fetchTasks = async () => {
    const res = await fetch("/api/tasks");
    setTasks(await res.json());
  };

  const addTask = async () => {
    const res = await fetch("/api/tasks", {
      method: "POST",
      body: JSON.stringify({ title, priority })
    });
    setTasks([...tasks, await res.json()]);
  };

  const toggleTask = async (id) => {
    const task = tasks.find(t => t.id === id);
    await fetch(`/api/tasks/${id}`, {
      method: "PATCH",
      body: JSON.stringify({ completed: !task.completed })
    });
    fetchTasks();
  };

  // Filter and render with priority colors and animations
}

    Navigate to http://localhost:3000/ to see your app running with hot reload included.

    ℹ️
    Zero Config Frontend
    Torque compiles React, serves bundles, and includes hot reload. No webpack, no build step. Just write React.

    Connect via Events

    Categories listen to task creation

    Categories subscribe to task events and update counts automatically:

    javascript bundles/categories/logic.js 
    export const setupSubscriptions = async (bus, db) => {
  bus.on("tasks.task.created", async (event) => {
    // Update task_count when task created
  });
};

export const createCategory = async (req, res, db, bus) => {
  const { name, color, icon } = req.body;
  const id = crypto.randomUUID();
  await db("categories").insert({ id, name, color, icon, task_count: 0 });
  bus.emit("categories.category.created", { id, name });
  res.json({ id, name, color, icon, task_count: 0 });
};
    💡
    Events are One-Way
    Bundles broadcast events and subscribe to them. Perfect for notifications, audit trails, and loosely-coupled updates.
    7

    Connect via Coordinator

    Query across bundles

    Tasks call the coordinator to fetch category details and enrich responses:

    javascript bundles/tasks/logic.js (updated) 
    export const listTasks = async (req, res, db, bus, coordinator) => {
  const tasks = await db("tasks").select("*");

  // Enrich with category details
  const enriched = await Promise.all(
    tasks.map(async (task) => {
      if (task.category_id) {
        const category = await coordinator.call(
          "categories",
          "getCategory",
          { id: task.category_id }
        );
        return { ...task, category };
      }
      return task;
    })
  );
  res.json(enriched);
};
    ℹ️
    Three-Guard Safety
    Coordinator checks: (1) bundle exists, (2) interface is exposed, (3) call fails gracefully if down. Your data is safe.

    Middleware & Hooks

    Add logging, auth, and audit trails

    Register hooks globally for logging, authentication, and audit trails:

    javascript torque.config.js 
    export default {
  hooks: [
    {
      event: "route:before",
      handler: (req, res, next) => {
        const start = Date.now();
        res.on("finish", () => {
          // Log with timing
        });
        next();
      }
    },
    {
      event: "event:after-publish",
      handler: (eventName, payload) => {
        console.log(`[AUDIT] ${eventName}`);
      }
    }
  ]
};

    Database Relationships

    Link data across bundles

    Tasks reference categories by ID. No SQL joins—coordinator fetches on demand:

    yaml bundles/tasks/manifest.yml 
    schema:
  task:
    id: uuid
    title: string
    completed: boolean
    priority: string
    due_date: datetime
    category_id: uuid  // Foreign key

    When listing tasks, the coordinator enriches each task with category details. Simple, safe, and loosely coupled.

    📌
    No Cross-Bundle Joins
    Bundles can't join tables. Instead, store the ID and use coordinator to fetch related data. This keeps bundles independently deployable.

    Your First Intent

    Declare what "done" looks like, not how to get there

    Intents are the heart of Torque's agentic-first design. Instead of writing controllers, you declare what "done" looks like. The kernel compiles each Intent into a dual-interface API — humans and agents use the same endpoint.

    An Intent is a triplet: Context (what data is needed), Behavior (safety guardrails), and the Intent contract itself (trigger, success criteria, description).

    Step 1: Generate the triplet

    Terminal
    $ torque generate intent support ResolveIssue

✓ Created bundles/support/intents/ResolveIssue/context.js
✓ Created bundles/support/intents/ResolveIssue/behavior.js
✓ Created bundles/support/intents/ResolveIssue/intent.js

    Step 2: Define Context

    The Context declares what data the Intent needs and how to vectorize it for semantic search:

    javascript bundles/support/intents/ResolveIssue/context.js 
    export default {
  schema: {
    issueId:      { type: "uuid",   required: true },
    customerId:   { type: "uuid",   required: true },
    description:  { type: "string", required: true },
    severity:     { type: "string", enum: ["low", "medium", "high", "critical"] },
  },

  vectorize(ctx) {
    return `Issue: ${ctx.description} (severity: ${ctx.severity})`;
  }
};

    Step 3: Define Behavior

    The Behavior sets guardrails — persona, allowed tools, and safety gates:

    javascript bundles/support/intents/ResolveIssue/behavior.js 
    export default {
  persona: "Senior Support Engineer with deep product knowledge",

  allowedTools: [
    "system_query",
    "knowledge_base_search",
    "send_customer_reply",
    "escalate_to_human",
  ],

  requireHumanConfirmation: [
    "send_customer_reply",   // gate dangerous actions
    "escalate_to_human",
  ],

  maxIterations: 10,
  timeoutMs: 30000,
};

    Step 4: Define the Intent

    The Intent contract ties Context and Behavior together with a trigger and success criteria:

    javascript bundles/support/intents/ResolveIssue/intent.js 
    import context from "./context.js";
import behavior from "./behavior.js";

export default {
  name:        "ResolveIssue",
  description: "Investigate and resolve a customer support issue",

  trigger: {
    event: "support.issue.created",
    condition: (ctx) => ctx.severity !== "critical",
  },

  successCriteria: [
    "Issue root cause identified",
    "Customer received a resolution or workaround",
    "Issue status updated to resolved",
  ],

  context,
  behavior,
};

    Step 5: Wire to Bundle

    Register your Intents in the bundle's logic.js via the intents() method:

    javascript bundles/support/logic.js 
    import ResolveIssue from "./intents/ResolveIssue/intent.js";

export const intents = () => [
  ResolveIssue,
];

    Step 6: The Dual-Interface

    Once wired, the kernel automatically registers the Intent as both a REST endpoint and an Agent Tool Schema. No extra code needed:

    Terminal
    $ torque dev

✓ support bundle loaded
✓ Intent registered: support.ResolveIssue
✓ REST endpoint: POST /api/intents/support/ResolveIssue
✓ Agent Tool Schema: support.ResolveIssue
✓ Server listening on http://localhost:3000

    Your React UI calls POST /api/intents/support/ResolveIssue with JSON. An AI agent calls the same endpoint. The kernel decides whether to invoke the LLM or handle it directly based on the payload. One endpoint, two interfaces, zero duplication.

    💡
    Safety is Structural
    requireHumanConfirmation physically gates dangerous tools at runtime. It is not a prompt instruction the model can ignore — the kernel enforces it before the tool executes. Safety by architecture, not by asking nicely.

    The Intent Trace

    Observe every step of Intent execution

    Every Intent execution emits a structured trace through the HookBus. You can observe, audit, and replay every decision the agent made:

    HookBus Output
    [INTENT: support.ResolveIssue] INITIATED by @michael
[INTENT: support.ResolveIssue] AGENT_THINKING
[INTENT: support.ResolveIssue] TOOL_INVOKED: 'system_query'
[INTENT: support.ResolveIssue] TOOL_INVOKED: 'knowledge_base_search'
[INTENT: support.ResolveIssue] HUMAN_CONFIRMATION_REQUIRED: 'send_customer_reply'
[INTENT: support.ResolveIssue] HUMAN_CONFIRMED by @michael
[INTENT: support.ResolveIssue] TOOL_INVOKED: 'send_customer_reply'
[INTENT: support.ResolveIssue] SUCCESS_CRITERIA_MET
[INTENT: support.ResolveIssue] RESOLVED

    The trace is not just logging — it is a first-class audit trail. Every TOOL_INVOKED records inputs and outputs. Every HUMAN_CONFIRMATION_REQUIRED records who approved and when. This makes Intent-Driven systems auditable by default.

    ℹ️
    Intent-Driven vs Controller-Driven
    Controllers tell a machine what to do. Intents tell it what "done" looks like. The framework compiles your Intent into a REST endpoint, an Agent Tool Schema, and real-time streams — you define constraints, not instructions.
    The language-agnostic, agentic-first composable monolith framework.
    You have now seen the full Torque development model: bundles for structure, events for communication, the coordinator for queries, hooks for cross-cutting concerns, and Intents for agentic-first development. Declare what "done" looks like. The kernel handles the rest.
